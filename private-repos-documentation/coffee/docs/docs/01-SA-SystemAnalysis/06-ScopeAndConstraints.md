### 1.2 排除範圍 (Out of Scope)

#### 明確不包含的功能
**Phase 1-3 都不包含**
- ❌ 線上訂餐系統 (顧客端 App/Web)
- ❌ 外送平台整合 (如 Uber Eats、Foodpanda)
- ❌ 會員行動 App
- ❌ 財務會計模組 (應收應付、總帳)
- ❌ 人力資源管理 (排班、薪資、考勤)
- ❌ 設備管理 (咖啡機、設備維護)
- ❌ 原料管理 (咖啡豆、牛奶等原物料追蹤)
- ❌ 多幣別支援 (僅支援 TWD)
- ❌ 多語系介面 (僅支援繁體中文)
- ❌ 硬體 POS 整合 (收銀機、條碼掃描器驅動)
- ❌ 生產製造管理
- ❌ 加盟商管理
- ❌ 電子發票系統整合 (API 對接)

#### 功能邊界說明
1. **僅門店端與總部端管理系統**
   - 不包含顧客端介面
   - 顧客無法自行線上訂購
   - 顧客無法查看個人會員資料

2. **簡化的財務功能**
   - 僅記錄營收金額
   - 不處理應收應付
   - 不產生財務報表
   - 不整合會計系統

3. **基本的庫存管理**
   - 以成品(商品)為單位
   - 不追蹤原物料
   - 不計算 BOM (Bill of Materials)
   - 不管理生產製程

4. **獨立系統**
   - 不與既有系統整合
   - 不對接外部 API (除支付服務)
   - 不整合硬體設備

## 2. 技術限制

### 2.1 開發技術限制

#### 必須使用的技術
- **後端**: Go 1.21+
- **後端框架**: Gin
- **前端**: Vue 3.x + TypeScript
- **前端框架**: Nuxt 3.x
- **資料庫**: MySQL 8.0+
- **快取**: Redis 7+
- **訊息佇列**: NATS 2.9+
- **容器化**: Docker, Docker Compose

#### 不使用的技術
- ❌ 其他後端語言 (Python, Java, Node.js 等)
- ❌ 其他前端框架 (React, Angular 等)
- ❌ NoSQL 資料庫作為主要儲存 (MongoDB 等)
- ❌ 其他訊息佇列 (RabbitMQ, Kafka 等)
- ❌ Kubernetes (Phase 1-2,使用 Docker Compose)

### 2.2 架構限制

#### 架構約束
- **微服務數量**: 8 個服務 (固定)
  - API Gateway
  - Auth Service
  - Customers Service
  - Orders Service
  - Inventory Service
  - Loyalty Service
  - Notifications Service
  - Reports Service

- **服務通訊**:
  - 同步通訊: HTTP/REST API
  - 非同步通訊: NATS (僅用於通知)
  - 不使用 gRPC

- **資料庫策略**:
  - 單一 MySQL 8.0+ 實例
  - 使用資料表前綴分隔不同服務的資料
  - 不使用分散式資料庫
  - Phase 1-2 不實施讀寫分離

### 2.3 效能限制

#### 系統容量限制
| 項目 | Phase 1 限制 | Phase 2 限制 | Phase 3 目標 |
|------|------------|------------|------------|
| 最大門店數 | 10 家 | 30 家 | 50 家 |
| 最大並發用戶 | 50 人 | 100 人 | 200 人 |
| 單日訂單量 | 5,000 筆 | 15,000 筆 | 25,000 筆 |
| 商品 SKU 數 | 100 個 | 300 個 | 500 個 |
| 活躍會員數 | 5,000 人 | 20,000 人 | 50,000 人 |
| 歷史資料保留 | 1 年 | 2 年 | 3 年 |

#### 效能指標限制
| 指標 | Phase 1 | Phase 2 | Phase 3 |
|------|---------|---------|---------|
| API 響應時間 (P95) | < 500ms | < 300ms | < 200ms |
| POS 下單完成時間 | < 5秒 | < 4秒 | < 3秒 |
| 報表產生時間 | < 10秒 | < 7秒 | < 5秒 |
| 系統可用性 | 99% | 99.5% | 99.9% |

## 3. 業務限制

### 3.1 使用者限制

#### 使用者數量
- **Phase 1**: 最多 50 個使用者帳號
- **Phase 2**: 最多 150 個使用者帳號
- **Phase 3**: 最多 300 個使用者帳號

#### 角色限制
- 固定 4 種角色: 系統管理員、總部管理、店長、店員
- Phase 1-2 不支援自訂角色
- Phase 3 可考慮彈性角色設定

### 3.2 資料限制

#### 單筆資料限制
- 訂單最多 50 個商品項目
- 採購單最多 100 個商品項目
- 單一商品最多 20 個規格
- 客戶標籤最多 10 個

#### 批次作業限制
- 單次匯入最多 1,000 筆資料
- 報表匯出最多 10,000 筆資料
- 客戶資料匯出需脫敏處理

### 3.3 地理限制

#### 支援地區
- **Phase 1-3**: 僅支援台灣地區
- 幣別: 新台幣 (TWD) only
- 時區: UTC+8 (台北時間)
- 稅制: 台灣營業稅 (5%)

#### 不支援地區
- 跨國連鎖店
- 多幣別交易
- 跨時區操作

### 3.4 合規限制

#### 必須遵守的法規
- **個資法 (PDPA)**
  - 客戶個資加密儲存
  - 資料存取記錄
  - 資料保留期限最多 3 年
  - 提供資料刪除功能

- **電子發票規範**
  - 發票格式符合台灣規範
  - 發票號碼管理
  - (Phase 1 僅紙本發票)

- **資料安全**
  - HTTPS 加密傳輸
  - 密碼雜湊儲存
  - SQL Injection 防護
  - XSS 防護

#### 不包含的合規要求
- PCI DSS (不自行處理信用卡資訊)
- 多國資料保護法規 (GDPR 等)

- ✅ HTTPS 加密傳輸

- ✅ 密碼 bcrypt 加密
- ✅ SQL Injection 防護

## 10. 風險與應對

### 10.1 範圍蔓延風險
**風險**: 需求持續增加導致範圍擴大  
**應對**: 
- 嚴格變更控制流程
- 新需求納入下一階段
- 優先級評估機制

### 10.2 技術限制風險
**風險**: 選定技術無法滿足需求  
**應對**:
- 技術預研 (PoC)
- 保留架構彈性
- 關鍵技術有備援方案

### 10.3 資源不足風險
**風險**: 人力或預算不足  
**應對**:
- 嚴格控制範圍
- 優先核心功能
- 靈活調整時程
